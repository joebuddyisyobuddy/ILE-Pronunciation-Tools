# ILE Phonetic Coach — Chrome Extension

Client-side pronunciation analysis that compares your speech against native-speaker reference audio. Runs entirely in the browser — no backend, no audio uploads, no server costs.

## Quick Start

1. **Load the extension**
   - Open `chrome://extensions` in Chrome/Edge/Brave
   - Enable **Developer mode** (top-right toggle)
   - Click **Load unpacked** → select the `ile-phonetic-extension/` folder
   - The 🎙 icon appears in the toolbar

2. **Test locally**
   - Open `test-page.html` in your browser
   - The Phonetic Coach panel appears in the bottom-right
   - (Optional) Place `.mp3` files in `test-audio/` to provide reference audio for the test slides

3. **Use on the ILE website**
   - Navigate to any lesson page on the ILE site
   - The extension auto-detects the current slide's text and audio
   - Click **▶ Reference** to hear the native recording
   - Click **⬤ Record** and speak the sentence
   - View your pitch, rhythm, and stress comparison instantly

## How It Works

The extension extracts acoustic features from both the reference recording and your recording, then compares them across four dimensions:

| Dimension | What it measures | Visualization |
|-----------|-----------------|---------------|
| **Pitch contour** | Rise and fall of fundamental frequency | Overlaid curves (blue = reference, orange = you) |
| **Energy / stress** | Where emphasis lands in the sentence | Mirrored amplitude envelope |
| **Rhythm** | Timing of syllable onsets | Timeline with onset markers |
| **Duration** | Overall speaking pace | Percentage difference |

Alignment is handled by **Dynamic Time Warping (DTW)**, which accounts for the fact that you speak at a different speed than the reference. The comparison is against the shape of the contour, not the absolute timing.

## Integrating With the ILE Website

The extension finds the reference text and audio by querying the page DOM. Update the selectors in `content.js` → `CONFIG.selectors` to match the actual ILE page structure:

```js
const CONFIG = {
  selectors: {
    // Element containing the sentence text
    sourceText: [
      '[data-source-text]',   // preferred: data attribute
      '.source-text',          // fallback: class name
      '.slide-text',
    ],

    // <audio> element with the reference recording
    sourceAudio: [
      'audio[data-reference]',
      'audio.reference-audio',
      'audio',                 // last resort: any audio on page
    ],

    // Container that changes when slide advances (for MutationObserver)
    slideContainer: [
      '.slide-container',
      '.lesson-container',
      'main',
    ],
  },
};
```

**Recommended ILE page markup** for cleanest integration:

```html
<div class="slide-container">
  <p class="source-text" data-source-text="The cat sat on the mat.">
    The cat sat on the mat.
  </p>
  <audio class="reference-audio" data-reference="true" preload="auto">
    <source src="/audio/lesson1/slide1.mp3" type="audio/mpeg" />
  </audio>
</div>
```

Selectors can also be configured per-user via the extension popup (click the toolbar icon).

## Architecture Decisions

**Why not Praat / Python?**
The extension knows exactly what sentence is being spoken. This constraint eliminates segmentation, phoneme identification, and all the hard parts of open-ended speech analysis. What remains — pitch tracking, energy, rhythm — is well within what the Web Audio API and a YIN implementation can handle at feedback-grade accuracy.

**Why DTW instead of forced alignment?**
Forced alignment requires a pronunciation dictionary and an acoustic model. DTW requires neither — it aligns two signals of the same utterance by finding the minimum-cost warping path. For known-sentence comparison, this is simpler and sufficient.

**Why no score?**
A score tells a learner nothing actionable. The feedback is dimensional: *your pitch rises where it should fall*, *your stress lands on syllable 2 instead of syllable 3*, *you're 30% slower than the reference*. Each dimension tells the learner what to change.

## Dependencies

Zero runtime dependencies. Everything uses browser-standard APIs:

- **Web Audio API** — recording, decoding, frequency analysis
- **MediaRecorder API** — microphone capture
- **Canvas API** — visualization rendering
- **Chrome Extension APIs** — storage, tabs

The pitch detection (YIN algorithm) and DTW alignment are implemented from scratch in the `lib/` folder. No npm, no build step, no bundler.

## Phoneme Recognition Setup

The extension can run a wav2vec2 neural network directly in the browser for true phoneme-level analysis. This requires a one-time model export:

```bash
pip install transformers torch onnx onnxruntime
python export_model.py
```

This downloads `facebook/wav2vec2-lv-60-espeak-cv-ft` from HuggingFace, exports it to ONNX format, and quantizes it to INT8 (~95MB). The result goes into `models/wav2vec2-phoneme-int8.onnx`.

When the extension loads and finds the model, it will:
1. Load ONNX Runtime Web (from CDN, ~2MB)
2. Download the model into IndexedDB (cached after first load)
3. Run inference locally on every recording
4. Output IPA phoneme sequences with timestamps
5. Diff them against the reference phonemes

If the model is not present, the extension still works — you just get acoustic analysis (MFCCs, formants, pitch) without phoneme labels. The model is purely additive.
